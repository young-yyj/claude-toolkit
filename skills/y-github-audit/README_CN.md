# GitHub Audit — 本地项目首次提交全维度审计

## 这是什么？

`y-github-audit` 是一个**仪表盘式** Skill，一次性对项目做 13 项并行扫描，汇总出一份健康报告，用户勾选要处理的问题，按优先级批量修复。

## 触发方式

在聊天中输入 **「YY本地项目初次提交github」** 即可启动。

> 不区分大小写，中英文混合也可以，比如「YY本地项目初次提交GitHub」。

## 它会做什么？

流程分 4 步，其中第 1 步的 13 项检查全部并行执行：

### Step 1 — 全维度并行扫描（13 项）

**安全扫描（S1-S5）：**
- S1：API Key / Token 明文检测（OpenAI、GitHub、Slack、AWS、JWT 等 8 种正则）
- S2：云服务凭证检测（AWS 配置、MongoDB/MySQL/PG/Redis 连接串、私钥头）
- S3：.env 文件 git 跟踪状态
- S4：私钥文件检测（.pem / .key / .pfx / .jks 等 7 种扩展名）
- S5：邮箱暴露检测（git config + git 历史 + 文件内容，三重检查）

**工程扫描（E1-E6）：**
- E1：.gitignore 存在性检查
- E2：技术栈检测（Node.js / Python / .NET / Go / Rust / Java / Ruby / PHP）
- E3：.gitignore 覆盖完整度检查（按技术栈对照必须忽略的项）
- E4：README 存在性与质量评估（字数 + 章节 + 双语）
- E5：分支状态（当前分支、最近提交、未暂存变更）
- E6：远程配置状态

**发布扫描（P1-P2）：**
- P1：版本号检测
- P2：综合可发布性判断

### Step 2 — 仪表盘汇总

产出分级报告：

- 🔴 严重问题（建议推送前处理）
- 🟡 建议处理
- 🟢 状态正常
- 🎯 发布建议

### Step 3 — 交互式选择处理

用户输入要处理的项序号（如 `1,2,4`），按优先级修复：**严重问题 → 建议问题 → 发布操作**。

每类问题的处理方式：

| 问题类型 | 处理选项 |
|---------|---------|
| API Key / Token 明文 | 替换为占位符 / 添加到 .gitignore / 跳过 |
| .env 被跟踪 | 添加到 .gitignore 并从索引移除 |
| 邮箱暴露 | 设置 GitHub 隐私邮箱 / 重写 git 历史 / 跳过 |
| 无 .gitignore | 根据技术栈自动生成 |
| .gitignore 缺漏 | 列出缺漏项，用户确认后补充 |
| README 质量低 | 展示具体问题，用户确认后补充 + 英文翻译 |

### Step 4 — 执行汇报

逐项汇报处理结果，给出 Release 和 push 的建议命令。

## 约束与安全红线

- 发现敏感信息时**先报告、等用户决策**，不自动修改
- 绝不执行 `git push`（含 `--force`、`--tags`），push 操作由用户亲自执行
- git 写操作（`commit`、`tag`）前必须给用户确认

## 适合谁用？

- 对 GitHub 发布流程熟悉的开发者
- 喜欢一眼看到全局、自己挑着修的效率型用户
- 只是想诊断项目、不一定要立刻 push 的场景

## 和 github-launch 的区别

| | github-launch | github-audit |
|------|--------------|-------------|
| 模式 | 线性向导，7 步顺序走 | 一次性并行扫描，仪表盘汇总 |
| 体验 | 导游模式，每步一个决策点 | 体检报告模式，勾选后批量修复 |
| 范围 | 含远程配置 + push + Release | 纯本地诊断 + 修复，不配远程 |

简单说：**Launch 是陪你走全程，Audit 是给你体检报告让你自己挑着修。**
