# Project Structure Standardizer — 项目结构规范化

## 这是什么？

`y-project-structure` 是一个**结构规范化** Skill，将任意项目目录对照 Claude Code 推荐的标准布局进行审计、迁移、补全。自动判定项目规模（小/中/大），输出差异报告，在用户逐项确认后创建缺失文件或迁移错位文件。

## 触发方式

以下任一表达均可触发（中英文混合亦可）：

- `整理项目结构` / `规范化目录` / `标准化项目`
- `项目结构优化` / `搭建 Claude 项目`
- `setup project structure` / `organize project`
- `帮我看看项目结构` / `这个项目目录有点乱`

## 它会做什么？

流程分三步，每步完成后向用户汇报再进入下一步。

### 第一步：项目评估

1. 统计源文件数量（`.cs`、`.js`、`.py` 等代码文件，不含工程文件和测试文件）
2. 统计测试文件数量
3. 检查是否存在多模块、多服务、多工作流等大型项目特征
4. 判定项目规模：

| 规模 | 判定条件 |
|------|---------|
| 小型 | 源文件 ≤ 5 |
| 中型 | 源文件 6-15 |
| 大型 | 源文件 > 15，或多服务/多工作流 |

不确定时默认按中型处理。

### 第二步：差异分析

对照 Claude Code 标准结构，逐项检查并输出结构化报告：

**匹配检查：**
- 列出已存在于标准位置的文件/目录
- 列出存在但位置不对的文件，给出建议迁移路径，并扫描工程中所有引用该文件路径的代码（import、require、using、项目文件引用等），列出需要同步更新的引用点

**缺失检查：**
- **基础必备**（强烈建议创建，即使项目当前没有相关文件）：`CLAUDE.md`、`.claude/settings.json`、`.gitignore`、`.env.example`、`docs/ai-context/`、`src/`
- **按需创建**（仅当项目已有对应类型文件时才创建）：`tests/`、`config/`、`scripts/`、`.claude/commands/`、`.claude/skills/`、`.github/workflows/`、`.mcp.json`、`CHANGELOG.md`、`LICENSE`

报告格式示例：

```
## 项目结构差异报告

### 已就位（N 项）
- ✅ CLAUDE.md
- ✅ .gitignore

### 需迁移（M 项）
- src/util.js → src/utils/

### 缺失 - 基础必备（X 项）
- ❌ .claude/settings.json
- ❌ .env.example

### 缺失 - 按需创建（Y 项）
- 💡 tests/（理由：项目已有 3 个测试文件）
- 🚫 config/（跳过：项目无配置文件）
```

### 第三步：执行

**所有创建和迁移操作均需用户逐项确认，不得静默执行。**

**基础必备项处理：**
- 逐项列出缺失项及理由，用户确认后逐项创建
- `CLAUDE.md`：信息提取优先级 README.md → 项目文件 → 留空
- `.claude/settings.json`：仅含 `permissions` 配置
- `.gitignore`：按标准模板创建
- `.env.example`：扫描项目中实际使用的环境变量列出键名

**按需创建项处理：**
- 满足触发条件的项建议创建，不满足的明确标注"跳过"及原因

**迁移执行（仅在用户明确要求时执行）：**

> ⚠️ **隔离原则**：涉及 `src/` 归位的迁移，先在隔离副本中完成全部操作并验证通过后，再写回原项目。原项目在验证通过前不得有任何改动。

1. **创建隔离副本**：git worktree 或临时目录
2. **在副本中迁移**：移动文件 + 更新所有路径引用
3. **四级验证**（自动排查 + 最多 3 轮重试）：
   - 编译/构建检查
   - 类型检查（如适用）
   - 测试运行
   - 启动验证
4. **写回**：验证通过后合并/覆盖回原项目

## 核心机制

### 两类目录

| 类型 | 示例 | 规则 |
|------|------|------|
| 基础必备 | `CLAUDE.md`、`.claude/settings.json`、`.gitignore`、`src/` 等 | 必须创建，即使当前为空也加 `.gitkeep` 保持可追踪 |
| 按需创建 | `tests/`、`config/`、`scripts/` 等 | 仅当项目已有对应类型文件或用户明确要求时才创建 |

### 内容填充原则

所有文件优先提取项目实际信息填写；信息不足时留空并添加注释说明本文件用途。**禁止编造不存在的信息。**

## 适用场景

- 新建项目需要搭建标准目录结构
- 现有项目目录混乱，希望规范化整理
- 不确定项目结构是否符合 Claude Code 最佳实践
- 想在迁移代码前了解所有受影响的引用点
