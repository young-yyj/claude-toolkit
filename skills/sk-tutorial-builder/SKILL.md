# Tutorial Builder — AI 领域教程生成 Skill

## 功能说明

针对指定的 AI 领域知识点，自动：
1. 全网多源搜集高质量资料（官方文档、论文、中文博客、微信公众号）
2. 交叉验证核心概念，去重整合
3. 按"从入门到精通"结构生成完整 Markdown 教程
4. 保存到项目本地文件并呈现给用户

---

## When to Use

- 用户要求生成 AI 领域知识点的教程、学习指南、入门到精通
- 需要系统性地整理某个 AI 技术方向的知识体系
- 需要一份包含原理、代码、实战的完整学习资料

## When NOT to Use

- 用户只需要简短的概念解释或名词定义（应直接回答）
- 用户只需要大纲/目录，不需要完整内容
- 主题非 AI 领域（此 Skill 专为 AI 领域设计）
- 用户明确要求输出格式非 Markdown

---

## Constraints

**内容质量要求：**
- 总字数 ≥ 20000 字
- 每个核心概念必须有代码示例
- 专业术语首次出现时给出英文原文
- 所有引用必须注明来源 URL（Markdown 链接格式），不得输出来源不明的数据或代码
- 教程风格必须为教科书式严谨连贯，杜绝碎片化与炒作式表达

**结构要求：**
- 每章开头必须显式标注分类标签（`[基础篇]` / `[进阶篇]` / `[高级篇]`）
- 教程开头格式统一为：
  ```
  > 更新日期：{YYYY-MM-DD}
  > 适用版本：{版本信息}
  ```
  - 更新日期来自 Step 1 中 `date +%Y-%m-%d` 的结果
  - 适用版本由 Claude 根据知识点当前主流版本自行判断（如 Python 3.12、PyTorch 2.5 等）；若不确定则向用户确认
- 必须先生成高质量大纲，再展开全文，严禁无结构一次性输出

**流程约束：**
- 若主题术语存在歧义，必须向用户提问确认，不得擅自臆断
- 必须搜索微信公众号内容（使用 `web_search` + `site:mp.weixin.qq.com`），不可跳过

## Do This / Not This

### 教程内容结构对比

Do This ✅：分层递进，基础→进阶→高级
```
[基础篇] 一、概述与背景
[基础篇] 二、核心原理
[进阶篇] 三、基础用法
[进阶篇] 四、进阶技巧
[高级篇] 五、高级应用
[高级篇] 六、前沿进展
```

Not This ❌：平铺直叙，缺少层次划分
```
一、介绍
二、什么是XX
三、更多内容
四、总结
```

### 代码示例对比

Do This ✅：完整可运行，含必要注释
```python
# Python 3.10+, 需要安装 transformers>=4.30
from transformers import pipeline

classifier = pipeline("sentiment-analysis")
result = classifier("I love this tutorial!")
print(result)  # [{'label': 'POSITIVE', 'score': 0.999}]
```

Not This ❌：伪代码或骨架，不可直接运行
```python
# 使用模型进行分析
model = load_model("xxx")
result = model.predict(data)
print(result)
```

### 概念解释对比

Do This ✅：定义 + 英文原文 + 类比 + 适用场景

> **注意力机制（Attention Mechanism）** 是 Transformer 架构的核心组件。可以将其理解为"在阅读文章时，大脑会自动聚焦在关键词语上"——模型在处理每个词时，会动态计算与其他词的相关权重。

Not This ❌：只有定义，缺少上下文

> 注意力机制是一种让模型关注重要信息的技术。

## 执行流程（简版）

### Step 1：分析知识点
- 首先执行 `date +%Y-%m-%d` 获取当前日期，用于教程 header
- 读取 `references/quality_guide.md`，了解来源评级和搜索策略
- 确定知识点类型、受众层次、前置依赖、提取搜索关键词（中英文各 5 个）

### Step 2：并行多源搜集

使用 **Agent** 工具并行执行 A/B/C 三类搜索，三类搜索之间无依赖关系，同时启动以提升效率。

**搜索记录要求**：记录每类搜索使用的搜索词、命中数和最终采用的来源数，便于质量回溯。

**最低信息量要求**：至少 8 个有效来源，S/A 级来源不少于 3 个。

**A. 官方文档 & 学术来源**（`web_search` + `web_fetch`）
- 官方文档、arXiv、GitHub、英文技术社区
- 参考 `references/quality_guide.md` 中的英文搜索词模板

**B. 中文技术博客**（`web_search`）
- 知乎 / CSDN / 掘金
- 参考 `references/quality_guide.md` 中的中文搜索词模板

**C. 微信公众号**（`web_search`，使用 `site:mp.weixin.qq.com` 限定）
- 加时间参数（如 `2025`、`2026`），搜索最近 1-2 年内的内容
- 目标：找到 5-10 篇高质量深度文章
- 搜索词参考 `references/quality_guide.md`


**兜底策略**：若某来源类型搜索不到足够内容，按以下顺序递补：
- 放宽时间范围（从 1 年扩至 2-3 年）
- 使用更宽泛的关键词重新搜索
- 用英文来源补充中文主题（或反之）
- 若仍不足 8 个有效来源，在教程中标注"该领域公开资料有限"

### Step 3：内容整合
- 核心概念多源交叉验证
- 代码示例筛选（可运行优先）
- 参考 `references/quality_guide.md` 的去重策略
- **为每个来源标注可信度评级**（S/A/B/C 级），高可信来源优先采用，低可信来源需交叉验证

### Step 4：选择大纲模板
读取 `references/outlines.md`，使用通用模板并根据类型差异速查表调整：
- 算法/模型 → 第二章增加数学基础、变体与改进
- 框架/工具 → 第二章改为"核心架构与API"，增加生产部署
- 概念/范式 → 第一章改为"概念解析"，增加效果优化
- 基础设施/MLOps → 第一章改为"背景与需求"，增加监控与运维
- **跨类型知识点可融合多个类型的调整要点，不必拘泥于单一类型**
- **建议增加"常见问题与踩坑经验"章节**（如适用），提升教程实用性
- **建议在末尾增加"总结与展望"章节**，收束全文并指出发展方向

### Step 5：生成教程

**前置确认**：回顾上方 `## Constraints` 章节中的全部要求，确保生成时严格遵循。

**章节写作顺序**：按大纲从第一章到最后一章顺序生成，确保前后连贯。

### Step 5b：输出前自检

生成完成后，对照上方 `## Constraints` 章节逐条检查，重点确认：

- 字数 ≥ 20000 字（使用 `wc -m` 精确验证）
- header 日期为当前日期
- 不存在占位符、未完成章节或重复内容
- 所有参考来源均已在文末列出

发现问题则修正后再进入 Step 6。

### Step 6：输出

1. 文件名：`{知识点}-从入门到精通-教程.md`，保存至项目本地目录（当前工作目录）
   - 知识点名称中含 `/ \ : * ? " < > |` 等特殊字符时，替换为全角字符或下划线，确保文件名合法
2. 执行 `wc -m "{文件名}"` 确认文件写入成功且字数达标
3. 将文件内容直接呈现给用户
4. 告知用户文件保存路径和字数
