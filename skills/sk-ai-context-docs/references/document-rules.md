# AI 上下文文档结构与质量规则

本文件为 `sk-ai-context-docs` 的参考规则。执行 skill 时先读取本文件，再扫描目标项目。

## 1. 适用范围

适用于需要沉淀 AI 上下文资料的项目，尤其是：

- 源码较多、目录较深的中大型项目。
- 已有 `CLAUDE.md`、`README.md`、架构说明或构建脚本的项目。
- 需要多次由 AI 辅助维护、排障、重构或扩展的项目。
- 需要把人类文档、构建配置、源码入口和模块结构整理为 AI 可快速读取资料的项目。

不负责：

- 编造项目背景、业务角色、部署环境或外部系统。
- 替代人工补充项目中不存在的信息。
- 移动项目目录或重构源码。
- 替代 `CLAUDE.md`、`README.md` 或人类架构文档。
- 在没有项目依据时生成完整但空泛的模板文档。

## 2. 来源与优先级

所有事实必须来自：项目内文档、`CLAUDE.md`、`README.md`、`AGENTS.md`、源码、构建文件、配置文件、脚本文件、资源清单、测试文件，或用户在当前任务中明确确认的信息。

读取优先级：

1. 项目内 AI 指南：`CLAUDE.md`、`AGENTS.md`、现有 `docs/ai-context/`。
2. 项目入口文档：`README.md`、中文项目说明、架构梳理文档。
3. 构建与工程文件：`.pro`、`.sln`、`.csproj`、`package.json`、`CMakeLists.txt`、`Makefile` 等。
4. 启动入口：`main.cpp`、`main.ts`、`App.*`、服务入口文件等。
5. 配置与资源清单：`.qrc`、`.env.example`、配置目录、平台配置头文件等。
6. 主要源码目录。
7. 测试目录、测试工程或调试页面。
8. 脚本目录和 CI/CD 工作流。

若多个来源冲突，优先信任当前源码和构建文件，并在输出总结中说明冲突来源；不要静默选择一个看似合理的结论。

## 3. 规模判定

规模只决定 `docs/ai-context/` 内生成哪些文件和子目录，不改变输出根目录。

| 规模 | 判定依据 | 建议产物 |
|---|---|---|
| 小型 | 源码少、目录浅、模块边界简单 | `README.md`、`00-quick-start.md`、`directory-map.md` |
| 中型 | 有明确模块、构建配置和运行流程 | 标准文档集合 |
| 大型 | 源码多、模块多、平台多、流程复杂 | 标准文档集合 + `modules/` + `flows/` + `decisions/` + `checklists/` |

规模判定不只看文件数量，也要结合项目结构、构建清单和文档描述。

## 4. 输出结构

### 4.1 最小结构

```text
docs/
  ai-context/
    README.md
    00-quick-start.md
    directory-map.md
```

### 4.2 标准结构

```text
docs/
  ai-context/
    README.md
    00-quick-start.md
    project-overview.md
    architecture.md
    directory-map.md
    build-run-test.md
    risk-areas.md
```

### 4.3 完整结构

```text
docs/
  ai-context/
    README.md
    00-quick-start.md
    project-overview.md
    architecture.md
    directory-map.md
    build-run-test.md
    risk-areas.md
    glossary.md
    modules/
      README.md
    flows/
      README.md
    decisions/
      README.md
    checklists/
      before-editing.md
```

只有当项目中存在可确认的模块、流程、术语、决策或检查项时，才生成对应文档。`decisions/` 仅在项目已有 ADR、架构决策记录、变更决策说明或用户明确确认的决策来源时生成；不得为了凑齐结构创建无内容或泛泛而谈的文件。

## 5. 文档内容规范

### 5.1 通用顶部格式

```md
# 文档标题

本文档用于说明……

## 信息来源

- `项目文件路径`
```

正文中的每个路径、命令、类名和流程都应能追溯到“信息来源”列出的项目文件。

### 5.2 `README.md`

作为 `docs/ai-context/` 入口索引，说明读取顺序、文档分工和使用规则。

必须来自项目的内容：

- 已生成文档列表。
- 推荐读取顺序。
- 项目内确认的高风险规则。

### 5.3 `00-quick-start.md`

作为 AI 的快速入口，只写最关键事实。

建议包含：

- 项目一句话说明。
- 主源码目录。
- 启动入口。
- 构建入口。
- 关键模块入口。
- 修改前必须阅读的文档。
- 修改后可执行的验证方式。

### 5.4 `project-overview.md`

说明项目用途、技术栈、运行环境和主要能力。所有内容必须来自项目文档、构建文件、源码或用户确认。

### 5.5 `architecture.md`

说明系统如何协作，不只列目录。优先使用项目文档、入口文件、核心类、构建清单和可追溯调用关系。

### 5.6 `directory-map.md`

用于帮助 AI 找文件。每行至少包含：

| Path | Purpose | Source |
|---|---|---|
| 项目真实路径 | 该路径在项目中的职责 | 信息来源文件 |

### 5.7 `build-run-test.md`

只写项目中可确认的命令。命令来源可以是 README、CLAUDE.md、构建脚本、工程文件或用户确认。无法确认的运行方式不写成命令。

### 5.8 `modules/*.md`

每个模块文档必须能对应真实目录、真实文件或构建清单中的文件组。

建议包含：

- 模块职责。
- 关键文件。
- 主要类或函数。
- 数据流。
- 修改规则。
- 风险。
- 验证方式。

没有足够来源时，不生成该模块文档。

### 5.9 `flows/*.md`

每个流程必须能从项目入口、调用链、信号槽连接、配置、脚本或现有文档追溯。

建议包含：

- 触发条件。
- 步骤。
- 关键文件。
- 状态变化。
- 失败情况。
- 验证方式。

没有足够来源时，不生成该流程文档。

### 5.10 `risk-areas.md`

风险必须来自项目本身，例如：

- 项目文档中明确提示的高风险区域。
- 构建清单中集中出现的核心模块。
- 平台配置、设备通信、数据存储、资源清单、升级脚本等可观察事实。

不能凭行业经验添加项目中无法确认的风险。

### 5.11 `checklists/*.md`

检查清单必须来自项目已有规则、构建要求、文件更新约束或用户确认。不生成泛用清单。

## 6. 人工确认点

遇到以下情况请求用户确认：

- 项目结构复杂，无法确定主源码目录。
- 构建命令存在多个候选。
- 项目有设备、嵌入式、外部服务或人工验证步骤，但项目中没有完整说明。
- 文档结构将生成较多文件。
- 源码与文档描述冲突。
- 用户要求写入项目中不存在的信息。

确认问题应具体，不应泛泛询问“是否继续”。

## 7. 质量门禁

写入完成前必须检查：

- [ ] 文档事实均来自项目本身或用户确认。
- [ ] 若项目根目录已有 `docs/ai-context/`，已读取既有文档并结合当前项目实际完成核对。
- [ ] 既有文档中仍然准确的内容已按统一格式保留或转换，过期、无来源或无法验证的内容未写入正文。
- [ ] 项目中没有来源的信息未写入正文。
- [ ] 没有编造路径、命令、模块、依赖、业务背景或外部系统。
- [ ] 文档路径均真实存在，或是本次刚创建的文档路径。
- [ ] 构建、运行、测试命令均有项目来源。
- [ ] 模块职责可追溯到源码、构建清单或项目文档。
- [ ] 流程可追溯到入口、调用链、信号槽、配置、脚本或项目文档。
- [ ] 没有 `TODO`、`TBD`、空模板或占位段落。
- [ ] 默认语言文本为中文。
- [ ] 代码、路径、命令、类名、函数名和固定术语保持原文。
- [ ] 输出总结列出生成文件、信息来源、跳过项和限制。

## 8. 失败处理

### 8.1 信息不足

不生成完整结构。只生成能被项目来源支撑的最小文档集合，并在总结中说明跳过原因。

### 8.2 来源冲突

优先信任当前源码和构建文件，在总结中列出冲突点，不把冲突内容写成确定事实。

### 8.3 无验证命令

不编造验证方式。可写项目中已有的人工验证入口，或说明“项目中未发现自动化验证命令”。

### 8.4 无模块边界

目录或源码不能支撑模块拆分时，不生成 `modules/*.md`，只生成 `directory-map.md`。
