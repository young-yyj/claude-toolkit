# Project Structure Standardizer

将任意项目目录对照 Claude Code 标准结构进行审计、迁移、补全。

## 工作流程

分三步执行，每步完成后向用户汇报再进入下一步。

### 第一步：项目评估

先读取 `references/standard-structure.md` 了解标准结构，然后扫描当前项目：

1. 统计源文件数量（`.cs`、`.xaml`、`.js`、`.py` 等代码文件，不含 `.csproj`、`.sln`、`package.json` 等工程文件，不含测试文件和配置文件），不论当前位于哪个目录
2. 统计测试文件数量
3. 统计工程文件数量（`.csproj`、`.sln`、`package.json` 等，用于判断是否需归入 `src/`，不计入规模判定）
4. 检查是否存在多模块、多服务、多工作流等大型项目特征
5. 据此判定项目规模：

| 规模 | 判定条件 |
|------|---------|
| 小型 | 源文件 ≤ 5 |
| 中型 | 源文件 6-15 |
| 大型 | 源文件 > 15，或多服务/多工作流 |

不确定时默认按中型处理，因为中型结构覆盖了最关键的规范项。

### 第二步：差异分析

对照 `references/standard-structure.md` 中对应规模的结构，逐项检查：

**匹配检查（已有文件 → 标准位置）**：
- 列出已存在于标准位置的文件/目录
- 列出存在但位置不对的文件，给出建议迁移路径
- **对每条迁移建议，扫描工程中所有引用该文件路径的代码**（如 import、require、include、using (C#)、x:Class / Source (XAML)、.csproj 引用），列出需要同步更新的引用点

**缺失检查（标准要求 → 当前项目）**，按两类分别处理：

### 第一类：基础必备（强烈建议创建，即使项目当前没有相关文件）

这些是项目的基础设施，不依赖项目已有内容即可独立存在。

| 项 | 理由 |
|----|------|
| `CLAUDE.md` | Claude Code 运行必需的项目指南 |
| `.claude/settings.json` | 权限与工具配置 |
| `.gitignore` | 通用版本控制规范 |
| `.env.example` | 环境变量模板，安全最佳实践 |
| `docs/ai-context/` | AI 上下文文档目录（需在 CLAUDE.md 中引用） |
| `src/` | 源代码统一目录：源文件（含 `.csproj`、`.sln` 等项目工程文件）散落根目录或其它位置时，归入 `src/` 并同步更新所有路径引用 |

### 第二类：按需创建（仅当项目已有对应类型文件时才创建）

这些是容器型目录或可选项——项目没有对应文件时创建空目录/空文件毫无意义。

| 项 | 触发条件 |
|----|----------|
| `tests/` | 项目已有测试文件，或用户明确计划编写测试 |
| `config/` | 项目已有应用配置文件 |
| `scripts/` | 项目已有自动化脚本 |
| `.claude/commands/` | 项目已有或需要自定义 Slash 命令 |
| `.claude/skills/` | 项目已有或需要自定义技能 |
| `.github/workflows/` | 项目已有 CI/CD 工作流文件 |
| `.mcp.json` | 项目已配置 MCP 服务器 |
| `.claude/settings.local.json` | 项目需要本地配置覆盖 |
| `CHANGELOG.md` | 项目有版本发布历史且需要记录 |
| `LICENSE` | 项目有明确的许可证需求 |

输出一份结构化差异报告，格式如下：

```
## 项目结构差异报告

### 已就位（N 项）
- ✅ CLAUDE.md
- ✅ .gitignore
...

### 需迁移（M 项）
- src/util.js → src/utils/
- MainWindow.xaml → src/Views/
- App.config → config/development.json（需转换格式）
...

### 缺失 - 基础必备（X 项）
- ❌ .claude/settings.json
- ❌ .env.example

### 缺失 - 按需创建（Y 项）
- 💡 tests/（理由：项目已有 3 个测试文件，建议归入 tests/ 统一管理）
- 🚫 config/（跳过：项目无配置文件，无需创建空目录）
```

### 第三步：执行

**所有创建和迁移操作均需用户逐项确认，不得静默执行。**

**第一类（基础必备）处理**：
- 逐项列出缺失项及理由
- 用户确认后逐项创建
- **内容填充通用原则**：所有第一类文件均优先提取项目实际信息填写；若项目无相关信息可写，内容留空并添加一行注释说明本文件/目录的作用，后续由开发者按需补充。**禁止编造不存在的信息**
- 各文件特殊说明：
  - `CLAUDE.md`：信息提取优先级 README.md → 项目文件 → 留空
  - `.claude/settings.json`：仅含 `permissions` 配置，不添加其他字段
  - `.gitignore`：按 `references/standard-structure.md` 第四节模板创建
  - `.env.example`：扫描项目中实际使用的环境变量列出键名；无则留空加注释
  - `docs/ai-context/`：创建目录（含 `.gitkeep`），目录下文档同样遵循通用原则

**第二类（按需创建）处理**：
- 扫描项目已有文件，列出满足触发条件的项（如"项目已有 3 个测试文件 → 建议创建 `tests/`"）
- 对不满足触发条件的项，明确标注"跳过"及原因（如"跳过 `config/`：项目无配置文件"）
- 用户确认后逐项创建满足条件的项

**迁移执行**（仅在用户明确要求时执行）：

> **⚠️ 隔离原则**：涉及 `src/` 归位的迁移（源文件从根目录或其他位置移入 `src/`），必须先创建隔离副本，在副本中完成全部操作并通过验证后，再写回原项目。原项目在验证通过前不得有任何改动。

**执行流程**：

1. **创建隔离副本**：
   - 若项目是 git 仓库：使用 `git worktree` 创建临时工作树
   - 若项目非 git 仓库：将项目整体复制到临时目录（如 `/tmp/project-migration/`）
   - 所有后续操作均在隔离副本中进行，原项目文件不受任何影响

2. **在隔离副本中执行迁移**：
   - 将源文件及项目工程文件（`.csproj`、`.sln` 等）移入 `src/` 目录
   - 同步更新第二步中已扫描出的所有路径引用（import、require、include、using、项目文件中的路径引用等）

3. **在隔离副本中验证（自动排查 + 重试）**：
   运行以下四级验证，每步失败时自动诊断错误根因并尝试修复（如未更新的 import 路径、配置文件引用遗漏等），修复后重新验证，最多重试 3 轮：
   - **编译/构建检查**：运行项目的构建命令（如 `npm run build`、`go build`、`cargo build`、`dotnet build`），确保无编译错误
   - **类型检查**：若项目有类型系统（TypeScript、mypy 等），运行类型检查
   - **测试运行**：运行项目现有测试套件，确保全部通过
   - **启动验证**：如项目有启动/运行命令，确认能正常启动
   - 3 轮重试后仍未通过，停止自动修复，保留隔离副本供用户排查，**原项目保持未改动状态**

4. **验证通过后写回**：
   - 若使用 git worktree：提交变更并合并回原分支
   - 若使用临时目录：将变更覆盖写回原项目
   - 清理隔离副本

5. **兜底**：若项目无自动化验证手段（无构建命令、无类型检查、无测试），不执行迁移，仅列出建议迁移清单及路径变更明细供用户手动决策

## 创建规则

- **目录**：第一类目录（如 `docs/ai-context/`）即使为空也创建，加 `.gitkeep` 保持 git 可追踪。第二类目录（如 `tests/`、`config/`、`scripts/`）仅当项目已有对应类型文件或用户明确要求时才创建——不凭空造空目录
- **第一类文件内容填充**：所有第一类文件均优先提取项目实际信息；信息不足时留空并添加一行注释说明本文件用途，禁止编造。各文件例外：`settings.json` 仅含 `permissions` 配置；`.gitignore` 按标准模板创建
- **从不覆盖已有文件**：创建前检查文件是否存在，存在则跳过并告知用户
- **迁移由用户驱动**：迁移建议只在报告中呈现，实际移动和路径更新仅当用户明确要求时执行

## 参考文档

完整结构定义见 `references/standard-structure.md`，在执行第一步前必须读取。
