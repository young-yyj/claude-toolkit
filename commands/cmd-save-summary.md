# cmd-save-summary

以**知识优先**的方式总结本次对话并保存为 Markdown 文件，输出语言为**中文**。

> **核心理念**：几个月后你回查这份总结时，关心的不是"当时问了什么顺序的问题"，而是"当时得出了什么结论、学到了什么、产出了什么"。结论先行，操作过程退后。

> **总结范围**：覆盖本次对话从开始到结束的全部内容。若对话极长（>200 轮），聚焦主要决策、最终结论和关键操作，中间探索性交互可压缩为一两句概述。

## 主题提炼指引

从对话中提取 1-2 个核心关键词作为文件名和标题中的主题，优先选择出现频次最高或占据最多讨论时间的话题。

示例：`Claude Code 自定义指令创建`、`API 调试与性能优化`

## 输出模板

AI 应按以下结构生成 Markdown 文件并保存。模板中 `{}` 包裹或中文说明性文字为占位提示，生成时替换为实际内容。

```
# YYYY-MM-DD 对话总结 — 主题关键词

> **摘要**：用 1-2 句话概括本次对话的核心内容（< 50 字），让读者一眼确认是否是自己要找的那份总结。

## 决策与结论

对话中做出的主要决策及其理由。每条一项：

- **{决策内容}** — {决策理由}（若有备选方案被否决，简要说明为何不选）

## 知识点

从对话中提取的技术概念、工具用法、最佳实践等，按主题分组。分组排版如下：

### {分组名 1}

- **{知识点}** — {简要解释，1-3 句话}。若仅顺带提及未深入展开，末尾标注 [提及]。

### {分组名 2}

- **{知识点}** — {简要解释}

分组参考（有则列，无则跳过）：工具配置、编程概念、调试技巧、工作流、API/协议、安全、性能、其他。

> 进入本节的唯一门槛：日后可能会再遇到这个概念。是否在对话中深入讨论过不影响收录。

## 产物清单

本次对话的实际产出：

| 类型 | 路径/内容 | 说明 |
|------|----------|------|
| 新建文件 | path/to/file | 为什么创建 |
| 修改文件 | path/to/file | 改了什么 |
| 删除文件 | path/to/file | 删除原因 |
| 配置变更 | 配置项 | 变更内容与原因 |
| 依赖安装 | 包名 | 安装目的 |
| 内容生成 | 非文件的产出描述 | 用途 |

无实际产物则写"本次对话无文件或配置变更"。

## 待办

- [ ] {待办事项}（优先级：{高 / 中 / 低}）
- [ ] {待办事项}（优先级：{高 / 中 / 低}）

无则写"无"。
```

## 索引维护

每次保存总结后，自动维护 `summaries/INDEX.md`：

1. 读取 `summaries/INDEX.md`
   - 若不存在，新建文件并写入表头：`| 日期 | 主题 | 标签 | 文件 |` + 分隔线
2. 将本次总结作为新行追加到表格中
3. 所有行按日期倒序重排（最新在上）
4. 写回 INDEX.md

### INDEX.md 示例

```
# 对话总结索引

| 日期 | 主题 | 标签 | 文件 |
|------|------|------|------|
| 2026-04-28 | save_summary 优化 | 工具配置, 工作流 | 2026-04-28_1730_save_summary优化.md |
```

标签字段从当前总结的知识点分组和决策关键词中提炼 2-4 个中文标签，用逗号分隔。

## 保存规则

- 保存目录：项目根目录下的 `summaries/`
- 文件名格式：`YYYY-MM-DD_HHmm_主题.md`，时间为当前时间，主题使用中文
- 保存完成后输出：总结文件路径 + INDEX.md 路径

## 注意事项

- 输出中四个章节靠标题层级（`##`）区分，无需 `---` 分隔线
- "决策与结论"与"知识点"内容若出现重叠，优先归入"决策与结论"——知识点只收录决策以外的概念性内容
- 总体风格：简洁、结构化、便于日后检索
- 不记录：敏感信息（密钥、密码、令牌）、纯探索性失败的中间步骤、重复性工具调用堆栈
